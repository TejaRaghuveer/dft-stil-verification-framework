# Pad Agent Documentation

## Overview

The Pad Agent provides **Primary I/O Driver and Monitor** functionality for DFT (Design for Test) verification. It controls primary inputs (PI), observes primary outputs (PO), and supports bidirectional pads with configurable timing, drive strength, and response comparison.

---

## Components

| Component | Description |
|-----------|-------------|
| **pad_xtn** | Transaction class: signal name/type, data value, timing (setup/hold/delay), drive strength, slew rate, expected response. |
| **pad_driver_enhanced** | Drives PI with setup/hold relative to clock, stores and broadcasts expected outputs, supports combinational/sequential stimulus, tri-state, and activity logging. |
| **pad_monitor_enhanced** | Captures PO changes, records waveform and timing, compares to expected (immediate or end-of-test), detects X/Z, alerts on mismatches, and gathers statistics (transitions per pin, timing variance). |
| **response_collector** | Aggregates expected outputs from ATPG/sequences (via driver’s `ap_expected`) and compares them with monitored outputs (via monitor’s `ap_captured`). |
| **pad_config** | Central config: number of PIs/POs, signal list, timing constraints, response tolerance, logging level. |
| **pad_agent_enhanced** | Unified agent: driver, monitor, sequencer, response_collector, and pad_config. |

---

## Configuration (pad_config)

- **num_primary_inputs / num_primary_outputs / num_pads** – Counts and total pad count.
- **signal_names[] / signal_types[]** – Ordered list of pad names and types (INPUT/OUTPUT/BIDIR).
- **setup_time_ns / hold_time_ns / propagation_delay_ns** – Timing constraints.
- **response_match_tolerance** – Real-valued tolerance for response comparison (e.g. analog).
- **response_immediate_check** – 1 = compare each cycle in monitor; 0 = end-of-test comparison.
- **logging_detail_level** – 0 = minimal, 1 = normal, 2 = verbose.

Set via:

```systemverilog
uvm_config_db#(pad_config)::set(this, "pad_agent*", "pad_cfg", pcfg);
```

## Multi-Domain Handling

- **Clock domain**: The pad interface has a `clk` signal. The driver uses it for **setup/hold** when `use_clock_ref` is set and stimulus type is SEQUENTIAL. All timing offsets (setup_time_ns, hold_time_ns, timing_offset) are in the same time units as the reference clock.
- **Multiple clocks**: For multiple domains, use one pad interface (and agent) per domain, or a single interface where `clk` is the dominant clock for pad timing. For true multi-domain, instantiate separate pad agents and drive/monitor with domain-specific sequences.
- **JTAG / other drivers**: Pad stimulus can be generated by the same sequencer that runs JTAG or other sequences. Synchronization is sequence-level: e.g. a sequence that does “JTAG shift then drive pads” keeps ordering. No direct TMS/TCK in the pad agent; sync is via test/sequence design.

---

## Timing Synchronization with JTAG and Clock Drivers

- **Clock driver**: The pad interface’s `clk` should be driven by the same clock driver used for the design (e.g. `clock_driver_enhanced`). Drive the same `clk` net to both the DUT and `pad_if.clk` so that pad setup/hold is aligned with the design clock.
- **JTAG**: Pad patterns (e.g. from STIL/ATPG) may be applied in parallel or after JTAG load. Keep ordering in the test:
  1. Start clock and JTAG.
  2. Run JTAG sequences (load, capture, etc.).
  3. Run pad sequences that drive PI and expect PO at the right cycles.
- **Ordering**: Use a single sequencer or a virtual sequence that coordinates JTAG and pad sequences so that “drive PI at cycle N” and “expect PO at cycle N+M” align with the same time base (clock edges) used by the pad driver and monitor.

## Usage Summary

1. **Build**: Create `pad_config`, set signal list and timing; put in config_db as `"pad_cfg"`.
2. **Agent**: Use `pad_agent_enhanced` (active or passive). Connect driver sequencer to your pad (and/or JTAG) sequences.
3. **Expected responses**: Send `pad_xtn` with `signal_type = OUTPUT` and `expected_response` (and optional `compare_mask`) from sequences; driver stores and broadcasts via `ap_expected`; response_collector compares with monitor data.
4. **End-of-test**: If `response_immediate_check == 0`, call `monitor.check_expected_at_eot()` (e.g. from env or test) to run deferred comparison and timing variance.

## Files

- `pad_driver_enhanced.sv` – Driver and `ap_expected`.
